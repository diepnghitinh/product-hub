## Prompt
> admin/people currently does not work, fix it.

## Diagnosis
`AdminPeoplePage` fetches `useUsers({ limit: 200 })` → `GET /v1/users?limit=200`. The shared `PaginationDto.limit` is capped at `@Max(100)`, and the global `ValidationPipe({ transform, whitelist })` **rejects `limit=200` with a 400**. The list query fails, so the People table renders empty (no rows — the page "doesn't work"). Routing (`RouterModule` → `path: 'users'`) and the `{ statusCode, data }` envelope were both fine; the request was simply over the cap.

## Fix
Changed every list call that over-requested to the max allowed (`limit: 200 → 100`). Same latent bug existed in 3 other places (identical defect), all fixed:
- `features/admin/AdminPeoplePage.tsx` (the reported one — users list)
- `features/bugs/api.ts` (`useBugs` — the whole bugs board was 400ing too, `QueryBugDto` also extends `PaginationDto`)
- `features/bugs/BugDetailPage.tsx` (assignee/mentions user picker)
- `features/reports/ReportView.tsx` (owner-picker users, added earlier today)

`limit: 100` is the server's max per call; fine for current team sizes (larger tenants would need real pagination). Verified with frontend `tsc --noEmit`; no `limit > 100` remains anywhere.
