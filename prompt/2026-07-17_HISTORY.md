## Backend ‚Äî multi-value filters
- **`shared/utils/query-array.util.ts`** (new) ‚Äî `TransformQueryArray()` normalizes `?x=a`, `?x=a,b` and `?x=a&x=b` to `string[]`; blank ‚Üí `undefined`. **Backwards compatible**: a single value becomes a 1-item array, so `$in` ‚â° the old equality match for existing callers.
- `query-bug.dto` / `query-task.dto` ‚Äî status, severity, assigneeId, projectId, roadmapId, roadmapItemId ‚Üí arrays with `@IsEnum(..., { each: true })`; validation still rejects junk (`?severity=bogus` ‚Üí **400** ‚úì).
- Both repositories ‚Üí `{ $in: [...] }`. Bug search also gained the **regex escaping + `_id` match** the task repo already had.

### Two real bugs found by verifying against the API rather than trusting the code
1. **axios would have silently broken every multi-select.** Axios 1.x serializes arrays as `?status[]=a` by default, and this API's query parser doesn't read the bracket form ‚Äî `?severity[]=low&severity[]=medium` returned **all 4 bugs** (filter dropped). Fixed globally in `lib/api.ts` with `paramsSerializer: { indexes: null }` ‚Üí repeated keys (`?status=a&status=b`), which I'd verified works. No existing caller sends arrays, so nothing else changes.
2. **"Unassigned" filtered nothing.** Unassigned is stored as `''`, but `?assigneeId=` is indistinguishable from "filter not set" (and the transform drops blanks) ‚Äî it returned **all 3** bugs. Added a documented sentinel (`UNASSIGNED_QUERY = '__unassigned__'` + `resolveAssignees()`) that the repos map back to `''`; the frontend exports the matching `UNASSIGNED`.

### Verified ‚Äî migration (the risky one)

```
customise global task statuses -> add "On staging"
eng team BEFORE restart: todo,in-progress,done   (never stored its own)
--- restart ---
eng team AFTER  restart: todo,in-progress,done,staging   <- inherited, nothing lost
```

First run of this "failed" ‚Äî but the new process had died on `EADDRINUSE` while the
**old** one still served :3000, so the health check passed against stale code. Killed
by pid, re-ran, migration worked. Worth remembering: `pkill -f "nest start"` doesn't
reliably kill it; `lsof -ti:3000 | xargs kill -9` does.
