## Prompt
> `roadmaps/:id` backlog item modal — change the heading to the backlog item's name, and allow editing it inline.

The modal's heading was a static `Edit item` / `Add item`, with the real title in a separate `h-11` input at the top of the form body. Promoted the name **into** the heading and deleted the body field (it would have been a duplicate).

## Work done
- **`components/ui/Dialog.tsx`** — `title` widened `string` → **`ReactNode`** so a dialog can render a custom heading. String titles keep the exact old markup, so **every existing call site is untouched**. A node title can't supply the dialog's accessible name, so added **`titleLabel`** → rendered in an `sr-only` `DialogPrimitive.Title` (Radix requires a Title; without it the dialog loses its accessible name). Close button gained `shrink-0` so a long title can't squash it.
- **`RoadmapItemDialog.tsx`** — heading is now an `Input` styled as the heading (`text-base font-semibold`, `border-transparent` → `hover:border-input` / `focus-visible:border-input`, so the edit affordance appears on hover with **no layout shift** — the border is always there, just transparent). `-ml-2 px-2` keeps the text optically aligned with where the old `<h2>` sat while the hover background bleeds into the header padding.
  - The heading renders in the Dialog **header**, i.e. outside `<form id="rm-item">` — so it carries **`form="rm-item"`**, which keeps it in the form's submit + constraint validation (`required` still blocks Done and focuses the field) and makes **Enter save**. Same association trick the footer's Done button already used.
  - `titleLabel` still resolves to `Edit item`/`Add item`, so the dialog's screen-reader name didn't regress.
- **i18n** — new `roadmaps.itemTitlePlaceholder` ("Untitled backlog item"); the heading keeps `roadmaps.itemTitle` as its `aria-label`.

## Work done
- **`features/tasks/components/PickTaskDialog.tsx`** (new) — search box (debounced 300ms, no debounce helper existed) → `useTasks({ search })`; rows show status dot, title, **the task id** (first 8 chars, full id on hover), and the item it's currently linked to. Click → `useUpdateTask` writes the link, `['tasks']` invalidates so the panel refreshes, dialog closes. Tasks already on this item are filtered out. - **Backend `task.repository.ts`** — `search` only matched `title`/`description`, so **searching by id could never have worked**. Added **`{ _id: re }`** to the `$or` (`_id` is a uuid **string**, so a regex filter is valid). Also **escaped the regex input** — the search box takes free text, and an unbalanced `(` would otherwise throw on `new RegExp`.
- **i18n** — `tasks.pick`/`pickTitle`/`pickSearch`/`pickEmpty`/`pickNone`/`pickLinkedTo`/`pickUnlinked`/`pickMoveHint`/`pickAction`.

### 🐛 Pre-existing bug found + fixed
`useTasks` requested **`limit: 200`**, but the backend's shared `PaginationDto` caps `limit` at **`@Max(100)`** — so `GET /tasks` was returning **400** and the Tasks panel could never load. Every other hook in the app already used `limit: 100`. Fixed `features/tasks/api.ts` → `100`. **Confirmed against the running API:** `limit=200` → `HTTP 400`, `limit=100` → `HTTP 200`.

### Decision taken (flagged)
The picker lists **tenant-wide** tasks, including ones already linked to *another* backlog item — picking one **moves** it here. Each row shows its current link, and `tasks.pickMoveHint` says so. Chosen because it's visible and reversible, and because scoping to a project would stop a pasted id from resolving.

Ran the installed lib (**tailwind-merge 3.6.0**): `twMerge('h-9 w-9','size-7')` → `size-7` ✓, with `[&_svg]:size-4` preserved.
