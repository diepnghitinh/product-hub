## Prompt
> roadmap roadmaps/a39574c7-5eab-459d-b2d9-396dc71eeaec , when I remove a column, nếu có items thuộc nó, hãy hỏi để move chúng sang 1 column khác, trước khi xóa.
>
> (When I remove a roadmap column, if it still has items, ask me to move them to another column before deleting.)

Before this, removing a column in **Manage columns** (`⋯ → Manage columns`) just dropped it. Its items — which store the column's `key` in `item.phase` — became orphans that `KanbanBoard.itemsFor` silently swept into the **first** column (see the old `columnsHint`: "Removing a column moves its items to the first column."). So items landed somewhere the user never chose, with no warning.

## Work done
- **`features/roadmaps/components/RoadmapColumnsDialog.tsx`** (the change lives here — the dialog already owned column edits)
  - Now takes an **`items` prop** and keeps a `draftItems` working copy alongside the existing `draft` columns.
  - The trash button calls **`requestRemove(col)`**: if the column holds items (`draftItems.filter(phase === col.key)`), it opens a **`MoveItemsPrompt`**; if empty, it removes directly (unchanged behaviour).
  - **`MoveItemsPrompt`** — a second `Dialog` stacked over Manage columns. Copy: *"'Next' still has 2 items. Choose a column to move them to before removing it."* + a **`Select`** of the remaining columns (options rendered with a `DotLabel` colour dot, defaulting to the first). Buttons: **Cancel** / **Move & remove** (primary).
  - **`confirmRemove(destKey)`** reassigns those items' `phase` to the chosen column in `draftItems`, flips `itemsDirty`, then drops the column from `draft`. Nothing is persisted yet — consistent with the dialog's draft-then-Save model, so Cancel discards everything.
  - **`save()`** now persists **items first** (`useReplaceRoadmapItems`), *then* columns (`useReplaceRoadmapColumns`) via `mutateAsync` — so a mid-save failure can never leave items pointing at a column that's already gone. Loading state covers both mutations.
- **`features/roadmaps/RoadmapBoardPage.tsx`** — passes `items={items}` into `RoadmapColumnsDialog`.
- **i18n (`i18n/en.ts`)** — rewrote `roadmaps.columnsHint` ("Removing a column with items lets you move them to another column first.") and added `roadmaps.moveItemsTitle`, `moveItemsStillHas`, `itemSingular`, `moveItemsTrail`, `moveItemsTo`, `moveAndRemove`. (`t()` has no interpolation, so the count sentence is composed in JSX.)

No new colours — the prompt reuses `Dialog`, `Select`, `DotLabel`, and the primary `Button`, all already themed. `tsc --noEmit` clean.

## Prompt
> Tôi có thể resize image bên trong 1 editorjs
>

The rich-text editor (`RichTextEditor`, used for task + roadmap-item descriptions with `images`) used the stock `@editorjs/image` tool, which has **no resize**. Notably, `lib/editorjs.ts` already round-tripped an image `width` (`file.width` ⇄ `<img style="width:…">`) — dormant plumbing whose comment even said *"set by the resize handle."* So the persistence half existed; only the interaction was missing.

## Work done
- **`lib/editor/ResizableImageTool.ts`** (new) — a self-contained Editor.js block tool replacing `@editorjs/image`, wire-compatible with the existing `{ file:{url,width}, caption, withBorder }` block shape:
  - **Drag-to-resize handle** at the image's bottom-right → writes a **responsive `%`** width (of the block's content column) into `data.file.width`, clamped 10–100%, with a live size badge while dragging. `block.dispatchChange()` on release so the editor persists it.
  - **Keyboard resize** — the handle is a focusable `role="slider"`; ←/→ nudge ±5% (a11y).
  - Carried over from the stock tool: upload to workspace storage with a **base64 data-URL fallback** when none is configured, image **paste / file-drop** (`pasteConfig` + `onPaste`), and an editable **caption**. Added an **"Add border"** tune (the converter already supports `withBorder`).
- **`components/ui/RichTextEditor.tsx`** — dropped `@editorjs/image` + the old `mediaImageTool`; registered `image: ResizableImageTool`. (`compressImageFile`/`uploadMedia` now live in the tool.)
- **`styles/rich-text-editor.css`** — styles for the frame / handle / size badge / caption / border, all using existing design tokens (`--primary`, `--ring`, `--border`, `--foreground`) — **no new colours**.

## Prompt
> fix `{ "statusCode": 500, "message": "The specified bucket does not exist", "error": "NoSuchBucket", "path": "/v1/uploads" }`

Once storage **is** configured (S3/S3-compatible), uploading to a bucket that doesn't exist made the AWS SDK throw `NoSuchBucket`. That's not a Nest exception, so it surfaced as a raw **500**.

## Work done
- **`backend/src/infrastructure/storage/storage.service.ts`** — `uploadS3` now **provisions the bucket on first use**: on a `NoSuchBucket` `PutObject`, it runs `CreateBucketCommand` then retries the put once. AWS needs a `LocationConstraint` for every region but `us-east-1` (S3-compatible endpoints ignore it), so it's set only when there's no custom endpoint and region ≠ us-east-1. `BucketAlreadyOwnedByYou`/`BucketAlreadyExists` races are treated as success; if creation genuinely fails, it throws an **actionable `BadRequestException`** ("bucket … couldn't be created — create it in your provider, or fix Settings → Storage") instead of a 500.
